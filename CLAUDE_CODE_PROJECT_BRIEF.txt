Battery Insight Project Brief
=============================

What this project is
--------------------
Battery Insight is a full-stack battery monitoring system for a multi-module battery pack. It collects sensor data from ESP32 boards over WiFi, stores and serves telemetry through an edge receiver and backend API, and visualizes everything in a React dashboard with real-time updates.

Core architecture
-----------------
1. ESP32 sensing boards collect voltage and temperature data.
2. ESP32 boards send JSON over WiFi.
3. Raspberry Pi receiver validates and writes data to InfluxDB.
4. API server queries InfluxDB and exposes REST and WebSocket endpoints.
5. React dashboard consumes the API and shows live charts, modules, alerts, and logs.

Important note
--------------
The main architecture is WiFi-based from ESP32 to receiver. There is a CAN-related folder in the repo, but that should be treated as optional or legacy unless you explicitly want to use it.

Tech stack
----------
- Frontend: React 18, Vite, Recharts
- Backend API: Node.js, Express, WebSocket
- Database: InfluxDB 2.x
- Edge receiver: Python, Flask, InfluxDB client
- Firmware: ESP32 with PlatformIO and Arduino-style C++
- Containerization: Docker, Docker Compose
- Optional tooling: Python mock generator, optional CAN reader folder

Main folders and what they do
-----------------------------

bms-dashboard-react
- React dashboard for monitoring the battery system.
- Contains overview, cells and modules, wireless health, contactors, logs and events, settings, export panel, error boundary, data service, and mock data generator.
- This is the user-facing UI.
- It should support both mock data and real API data.
- It uses a modular component structure.
- Visual polish and real-time updates are important.

api-server
- Node.js backend that bridges InfluxDB to the dashboard.
- Responsibilities: health check, current state, historical telemetry, module history, alerts, export, contactors, stats, WebSocket live updates.
- This is the main read API for the dashboard.
- It queries InfluxDB directly.
- It uses retry logic and modular query helpers.
- It should stay lightweight and maintainable.

esp32-firmware
- Embedded firmware for ESP32 boards.
- Structure includes master firmware, sensing firmware, and PlatformIO project setup.
- Sensing boards collect sensor data and send it over WiFi.
- Master board handles contactor or system-level sensing.
- Board-specific settings are stored in config headers.
- WiFi SSID, password, host, module ID, and pin mappings are configured per board.

rpi-receiver
- Python receiver service for incoming ESP32 telemetry.
- Responsibilities: receive HTTP payloads, validate JSON, apply range checks, buffer and aggregate module data, write telemetry into InfluxDB, track per-module last-seen data and health.
- This is the edge ingestion layer.
- It is the primary bridge between hardware and the database.
- It is important for data validation and resilience.

mock-generator
- Python service that simulates telemetry.
- Useful for development, demos, and testing without hardware.

can-reader
- Python CAN-related bridge.
- This appears to be present in the repo, but the intended primary architecture is WiFi from ESP32s.
- Do not assume CAN is the main transport unless you want to reintroduce it.

Data model and telemetry concepts
---------------------------------
The project uses a time-series model in InfluxDB. Main measurements include:
- pack_metrics
- module_metrics
- cell_metrics
- wireless_health
- contactor_status
- module_health
- alerts

Common data fields include:
- voltage
- current
- power
- SOC
- SOH
- temperatures
- RSSI
- packet loss
- latency
- module health
- alert severity and thresholds

This means the database should be treated as time-series telemetry, not relational records.

Runtime behavior
----------------
Production:
- Docker Compose orchestrates services.
- InfluxDB persists telemetry.
- API server serves the dashboard.
- Dashboard is served through Nginx in a container.
- ESP32 boards post telemetry to the receiver.

Development:
- Dashboard can run locally with Vite.
- Mock generator can simulate live data.
- API server can run independently.
- Receiver can be run separately for hardware testing.

Architectural conventions
-------------------------
The codebase is structured around separation of concerns:
- Firmware collects data.
- Receiver validates and persists it.
- API server reads and transforms it.
- Frontend presents it.
- Mock tools simulate it.

Claude should preserve that layering and avoid mixing responsibilities across tiers.

What Claude Code should be careful about
-----------------------------------------
1. Do not reintroduce CAN as the primary data path.
2. Keep ESP32 communication over WiFi.
3. Preserve the modular dashboard component structure.
4. Keep InfluxDB as the source of truth for telemetry.
5. Keep the API server thin and focused on querying and formatting data.
6. Keep receiver-side validation and buffering strong.
7. Maintain support for mock data so the UI can be developed without hardware.
8. Avoid adding unnecessary dependencies unless they solve a real problem.

Best places to extend the system
--------------------------------
- Add more dashboard analytics in bms-dashboard-react.
- Add caching or rate limiting in api-server.
- Add more validation and batching in rpi-receiver.
- Add better alert logic and thresholds.
- Add historical trend and prediction features.
- Add configuration management for more ESP32 modules.
- Add automated tests for API and receiver logic.

Short one-paragraph summary
---------------------------
Battery Insight is a WiFi-based ESP32 battery telemetry platform with a Python edge receiver, InfluxDB time-series storage, a Node.js REST/WebSocket API, and a React dashboard. It is built to monitor multi-module battery packs in real time, show cell, module, contactor, and wireless health, and support both live hardware and simulated development workflows.
